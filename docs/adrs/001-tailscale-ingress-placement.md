# ADR-001: Tailscale Ingress Placement

**Status:** Accepted · **Date:** 2026-05-29

## Context

This homelab exposes several internal services via Tailscale using `Ingress` resources with `ingressClassName: tailscale`. These ingresses were scattered across different Helm charts:

- `platform/tailscale/templates/platform/` — ArgoCD, Vault, Nginx (disabled)
- `platform/monitoring/templates/` — Grafana, Prometheus

The question was whether all Tailscale ingresses should be centralized in the tailscale chart or remain co-located with their respective application charts.

An additional architectural consideration: the repo is structured as `platform/` (base infrastructure services) and `apps/` (user-facing applications). Any placement strategy should respect this boundary.

## Options Considered

### Option A — Co-located (status quo)

Each chart owns its own Tailscale ingress templates. The tailscale chart contains ingresses for services deployed before or alongside it; monitoring contains its own.

**Pros:** Each chart is self-contained. No cross-chart service name coupling.

**Cons:** Inconsistent pattern. No single view of "what's exposed via Tailscale". Sync-wave ordering concerns for pre-tailscale services (Vault in wave 1, tailscale operator in wave 2).

### Option B — Centralized in tailscale chart for platform, co-located for apps (SELECTED)

All platform services expose their Tailscale ingresses in `platform/tailscale/templates/platform/`. App services manage their own ingresses in their respective `apps/<app>/templates/` charts.

**Pros:**
- Single source of truth for platform-level Tailscale exposure
- Consistent pattern: same directory, same gating (`{{- if .Values.<service>.enabled }}`), same template structure
- Respects the platform/apps architectural boundary
- Sync-wave ordering is resolved: tailscale deploys in wave 2 (operator + all platform ingresses), monitoring in wave 3

**Cons:**
- The tailscale chart references service names from other charts (e.g., `monitoring-grafana`)
- Service name changes must be coordinated across charts

### Option C — Dedicated exposure chart

Create a separate `platform/exposure` chart with only Tailscale ingress templates, deployed after the tailscale operator.

**Pros:** Complete separation of concerns. Operator and exposure are independent.

**Cons:** Another chart to maintain. Adds coordination overhead disproportionate to homelab scale.

### Option D — Centralized for everything (platform + apps)

All Tailscale ingresses — platform and apps — live in the tailscale chart under `templates/platform/` and `templates/apps/` subdirectories.

**Pros:** Truly single source of truth.

**Cons:** Couples app exposure to an infrastructure chart. Adding a new app requires modifying the tailscale chart. Violates the platform/apps domain boundary.

## Decision

**Option B: Centralized for platform, co-located for apps.**

All platform-level Tailscale ingresses live in `platform/tailscale/templates/platform/`. Each is gated by its own `.Values.<service>.enabled` toggle in `platform/tailscale/values.yaml`.

Apps in `apps/` manage their own Tailscale exposure independently.

## Rationale

1. **Domain alignment.** Platform services (ArgoCD, Vault, Grafana, Prometheus) are the base of the system — they are always deployed and exposed the same way. Apps are independent domains; coupling them to an infrastructure chart creates unnecessary friction.

2. **Sync-wave ordering.** Tailscale deploys in ArgoCD wave 2 (operator + all platform ingresses). Monitoring deploys in wave 3. By moving the Grafana/Prometheus ingresses into the tailscale chart, they are created when the operator already exists — no orphaned ingress class and no health-check issues.

3. **Consistency.** Every platform service follows the same template pattern in the same directory with the same gating mechanism. The monitoring chart no longer needs its own `tailscale.enabled` toggle.

4. **Traceability.** One place to answer "what platform services are exposed via Tailscale?" — `platform/tailscale/templates/platform/`.

## Consequences

- **Positive:** Single view of platform exposure, consistent gating, resolved sync-wave ordering.
- **Negative:** `platform/tailscale/values.yaml` now references service names from other charts (e.g., `monitoring-grafana`). These names are stable (generated by kube-prometheus-stack) but must be kept in sync.
- **Migration:** Removed `platform/monitoring/templates/grafana-ingress.yaml` and `prometheus-ingress.yaml`. Removed `tailscale.enabled` from `platform/monitoring/values.yaml`. Added `grafana.enabled` and `prometheus.enabled` to `platform/tailscale/values.yaml`.
